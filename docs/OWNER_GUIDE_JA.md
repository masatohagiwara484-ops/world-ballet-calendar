## 0. 最初に：今日やることの地図（全体像）

オーナーの作業は大きく分けて **4つ** あります。順番が大事です。

```
①  Supabase に「SQL スキーマ」を流す（データの「箱」を作る）
        ↓
②  鍵（API キー）を .env.local に書き込む（パソコンとデータベースをつなぐ）
        ↓
③  ターミナルで seed → verify（箱の中にデータを入れて、正しいか確認）
        ↓
④  SOURCES（取材元リスト）の空欄を埋めて Claude Code に渡す（本物のデータの源を集める）

- **①②③** は「サイトを動かすための準備」。一度やれば基本おわりです。
- **④** は「これから継続的に集めていく素材」。あなたの一番の付加価値です。
- **Telegram bot / VPS** については → 第6章。**結論：今は使いません。**

> 💡 用語：**Supabase（スーパーベース）**＝インターネット上にある「データの保管庫（データベース）」。
> 公演データ（いつ・どこで・何の演目か）をここに貯めます。無料で始められます。

## 1. Supabase に SQL スキーマを流す（「SQL が見つからない」問題の解決）

### 1-1. そもそも「SQL スキーマ」とは？どこにある？

- **SQL（エスキューエル）**＝データベースに指示を出すための言葉。
- **スキーマ**＝「データをどんな形で保存するか」という設計図（テーブル＝表の定義）。
- あなたが探していた SQL ファイルは、このプロジェクトの中の以下の場所にあります👇

```
supabase/migrations/
 ├── 001_initial_schema.sql   ← 古い版（使いません）
 ├── 002_rebuild_schema.sql   ← ★これを最初に流す（基本の箱）
 └── 003_entity_graph.sql     ← ★次にこれを流す（横断検索用の箱）
```

> 💡 用語：**migrations（マイグレーション）**＝「データベースの設計変更ファイル」。
> 番号順（001→002→003）に実行していくのがルールです。
> このプロジェクトでは **002 と 003 の2つ**を実行すれば OK です（001 は飛ばします）。

### 1-2. 実行手順（コピペで完了。ターミナル不要・ブラウザだけ）

1. ブラウザで **https://supabase.com** にログインし、作成済みのプロジェクトを開く。
2. 左side のメニューから **「SQL Editor」**（紙とペンのようなアイコン）をクリック。
3. 右上の **「+ New query」**（新しいクエリ）ボタンを押す。
4. パソコンで `supabase/migrations/002_rebuild_schema.sql` を開き、**中身を全部コピー**する。
   - ファイルの開き方が分からない場合は、ターミナルで次を打つと中身が画面に出ます:
     ```bash
     cat supabase/migrations/002_rebuild_schema.sql
     ```
5. コピーした内容を SQL Editor の入力欄に **全部貼り付け**る。
6. 右下（または右上）の緑色の **「Run」** ボタンを押す。
7. 画面下に `Success. No rows returned` のような表示が出たら成功です。
8. **同じことを `003_entity_graph.sql` でも繰り返す**（New query → 貼り付け → Run）。

> ⚠️ 注意：**必ず 002 → 003 の順番**で実行してください。順番を逆にするとエラーになります。
> エラーが赤字で出たら、メッセージをそのまま Claude Code に貼り付けて相談してください。

### 1-3. うまくいったかの確認

左メニューの **「Table Editor」** を開き、`companies`, `performances`, `people`,
`works`, `venues` などの表（テーブル）が並んでいれば成功です。
（この時点では中身は空っぽ＝0行で正常。データは第3章で入れます。）

## 2. 環境変数を `.env.local` に保存する（鍵をパソコンに登録する）

### 2-1. 「環境変数」「.env.local」とは？

- **環境変数（かんきょうへんすう）**＝プログラムに渡す「秘密の設定値」。今回は Supabase の鍵など。
- **`.env.local`**＝その秘密を書いておく**ファイル**。ファイル名がドット（`.`）で始まるのが特徴。
  - `local`＝「自分のパソコンの中だけ」という意味。**GitHub には絶対アップロードされません**
    （`.gitignore` で除外設定済み）。だから鍵を書いても安全です。

> ⚠️ 超重要：このファイルの中身（特に下の `SERVICE_ROLE_KEY`）は**他人に絶対見せない**こと。
> SNS・GitHub・スクショに写さない。漏れると DB を勝手に書き換えられます。

### 2-2. 何の鍵を入れるのか（3つ）

Supabase の画面で **「Project Settings」（歯車アイコン）→「API」** を開くと取得できます。

| 環境変数の名前 | どこで取る | 役割 |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | API ページの「Project URL」 | DB の住所（公開してOK） |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | API ページの「anon public」キー | 読み取り用の鍵（公開してOK） |
| `SUPABASE_SERVICE_ROLE_KEY` | API ページの「service_role」キー（🔒目隠しを外して表示） | **書き込み用の鍵（極秘）** |

> 💡 `service_role` キーは「秘密にしてね」と Supabase 画面でも警告が出ます。
> seed（データ投入）で必要なので使いますが、**.env.local の中だけ**に置きます。

### 2-3. ターミナルで `.env.local` を作る（コマンド付き）

> 💡 用語：**ターミナル**＝パソコンに文字でコマンド（命令）を打ち込む黒い画面。
> Mac は「ターミナル.app」、Windows は「PowerShell」や「コマンドプロンプト」。
> このプロジェクトのフォルダを開いた状態で操作します。

**ステップ A：プロジェクトのフォルダへ移動**
```bash
cd /home/user/world-ballet-calendar
```
（自分のパソコンでは、フォルダを置いた場所のパスに置き換えてください。）

**ステップ B：`.env.local` を作って書き込む**

一番かんたんなのは、テキストエディタ（メモ帳・VS Code など）で新規ファイルを作り、
ファイル名を **`.env.local`** にして、下の3行を貼り付けて保存する方法です。

```dotenv
NEXT_PUBLIC_SUPABASE_URL=https://あなたのプロジェクトID.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=ここにanonキーを貼る
SUPABASE_SERVICE_ROLE_KEY=ここにservice_roleキーを貼る
```

- `=` の**前後にスペースを入れない**。
- 値を **引用符（" や '）で囲まない**。
- キーは長い文字列です。改行が入らないよう、1行に収めてください。

**ターミナルだけで作りたい場合**（コピペで実行）:
```bash
cat > .env.local << 'EOF'
NEXT_PUBLIC_SUPABASE_URL=https://あなたのプロジェクトID.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=ここにanonキーを貼る
SUPABASE_SERVICE_ROLE_KEY=ここにservice_roleキーを貼る
EOF
```
> 上の3行の「ここに〜貼る」を、自分の実際のキーに書き換えてから実行してください。

**ステップ C：ちゃんと書けたか確認**
```bash
cat .env.local
```
3行が表示されればOK（このコマンドは中身を画面に出すだけ。誰かに送られたりしません）。

### 2-4. GitHub 側に環境変数を設定する場合（GitHub Secrets）

> 現状あなたは **GitHub Secrets を使っていない**ので、**今すぐは不要**です。
> ただし将来、GitHub Actions（自動でデータ取得を動かす仕組み）を使うときに必要になります。
> そのときの操作位置を、迷わないよう先に書いておきます。

1. ブラウザで GitHub のリポジトリページを開く:
   `https://github.com/masatohagiwara484-ops/world-ballet-calendar`
2. 上のタブの一番右 **「Settings」（歯車）** をクリック。
3. 左メニューを下にスクロールし **「Secrets and variables」** をクリック → **「Actions」** を選ぶ。
4. 緑のボタン **「New repository secret」** を押す。
5. **Name** に環境変数名（例：`SUPABASE_SERVICE_ROLE_KEY`）、**Secret** に値を貼る → **「Add secret」**。
6. 必要な数だけ 4〜5 を繰り返す（`NEXT_PUBLIC_SUPABASE_URL` なども同様）。

> 💡 GitHub Secrets ＝「GitHub のサーバー上で安全に鍵を保管する金庫」。
> Vercel（本番サイト用）には既に設定済みとのことなので、**ローカル作業は `.env.local` だけで進められます**。

## 3. seed ＋ verify（データを入れて、正しいか確認する）

### 3-1. これはターミナル操作です（ブラウザではできません）

- **seed（シード＝種まき）**＝用意済みの公演データを Supabase の箱に流し込む作業。
- **verify / validate（検証）**＝データに矛盾がないか自動チェックする作業。

第1章（SQL スキーマ）と第2章（.env.local）が**先に終わっていること**が条件です。

### 3-2. 事前準備：必要なソフトと部品をそろえる（初回だけ）

> 💡 用語：**Node.js（ノード）**＝このプロジェクトを動かす土台のソフト。
> **npm（エヌピーエム）**＝Node.js に付属する「部品の取り寄せ係」。

**Node.js が入っているか確認:**
```bash
node -v
```
`v18` や `v20` のような数字が出ればOK。`command not found` と出たら
https://nodejs.org から「LTS」版をインストールしてください。

**プロジェクトの部品（ライブラリ）を取り寄せる（初回だけ、数分かかります）:**
```bash
cd /home/user/world-ballet-calendar
npm install
```

### 3-3. 実行コマンド（順番に1行ずつ）

**① まずデータの中身が正しいか検証（DBに触らない安全なチェック）:**
```bash
npm run validate:data
```
- 緑色で問題なしと出れば成功。赤いエラーが出たら内容を Claude Code に貼って相談。

**② データを Supabase に投入（seed）:**
```bash
npm run seed
```
- `.env.local` の鍵を自動で読み込み、companies / performances を流し込みます。
- 「Seeded N companies / M performances」のような成功メッセージが出れば完了。

**③ ブラウザで最終確認:**
- Supabase の「Table Editor」で `companies` テーブルを開き、行（劇場のデータ）が入っていればOK。
- もしくは開発サーバーを立ち上げてサイトで確認:
  ```bash
  npm run dev
  ```
  表示された `http://localhost:3000` をブラウザで開く。終了は `Ctrl + C`。

### 3-4. よくあるエラーと対処

| エラーの内容 | 原因 | 対処 |
|---|---|---|
| `Missing Supabase env vars` | `.env.local` が無い／名前ミス | 第2章をやり直す。`cat .env.local` で確認 |
| `placeholders — refusing to seed` | キーが仮の値のまま | 本物のキーに差し替える |
| `relation "companies" does not exist` | SQL スキーマ未実行 | 第1章の 002→003 を流す |
| `command not found: npm` | Node.js 未インストール | 3-2 を実施 |

> 💡 **次回からは①→②の2コマンドだけ**で更新できます。準備（3-2）は初回のみです。

## 4. SOURCES（取材元リスト）の埋め方 — 4つの列の意味

`docs/SOURCES.md` には劇場（バレエ団・オペラ団）の一覧があり、各団体ごとに
**4つの空欄（TODO）**を埋めるのがオーナーの仕事です。「何のリンクを貼ればいいか分からない」を
ここで完全に解消します。**コードを書く必要は一切ありません。調べて貼るだけです。**

### 4-1. 各列に「何を貼るか」早見表

| 列の名前 | 何を貼る？ | 探し方のコツ | 記入例 |
|---|---|---|---|
| **Listing URL**（公演一覧ページ） | トップページではなく「シーズン／演目一覧」ページの**正確なURL** | 公式サイトで `Season`・`What's On`・`Calendar`・`公演情報` のリンクを探す | `https://www.rbo.org.uk/whats-on` |
| **Feed**（機械が読める形式） | 自動取得できるデータの**URL**。優先順位 `API ▸ iCal ▸ RSS ▸ HTML` | 「カレンダー登録」「iCal」「RSS」ボタンを探す。無ければ `HTML` と書く | `iCal: https://.../events.ics` または `HTML` |
| **Affiliate**（チケット販売／提携） | その劇場の**チケット購入リンク**、または提携サービス名 | 「Book tickets」リンク。Tiqets/GetYourGuide で扱いがあるか | `direct: https://.../tickets` / `Tiqets` |
| **robots/ToS**（自動取得の可否） | 自動アクセスが許可されているかの判定：`OK` / `restricted` / `forbidden` | `公式URL/robots.txt` をブラウザで開いて確認（4-3参照） | `OK` |

### 4-2. 各列の「意味」をもう少しやさしく

- **Listing URL**：私たちのプログラムが「毎日見に行く」ページ。だから**トップではなく公演一覧**。
  - 💡 例：`operadeparis.fr`（トップ）ではなく `operadeparis.fr/en/season-and-tickets/season` のような一覧ページ。
- **Feed**：劇場が用意してくれている「データの蛇口」。
  - 💡 用語：**iCal（アイカル）**＝カレンダーアプリに取り込める予定ファイル（`.ics`）。
    **RSS**＝更新情報を配信する仕組み。**API**＝データを直接くれる窓口。
  - これがあると、サイトの見た目が変わっても壊れません。**最重要列**です。無ければ正直に `HTML`。
- **Affiliate**：将来チケット経由で収益を得るための情報。今は**チケット購入ページのURL**を貼れば十分。
  分からなければ空欄でなく `unknown` と書いてOK。
- **robots/ToS**：法律・マナーの確認。**自動でページを読みに行ってよいか**の判断材料。

### 4-3. robots/ToS の確認方法（コマンド不要・ブラウザだけ）

1. その劇場の公式URLの末尾に **`/robots.txt`** を付けてブラウザで開く。
   - 例：`https://www.rbo.org.uk/robots.txt`
2. 表示されたテキストの中で、対象の公演ページが `Disallow`（禁止）になっていないか見る。
   - 全部許可っぽい／公演ページが禁止されていない → `OK`
   - 公演ページが `Disallow` で禁止 → `forbidden`
   - 一部だけ条件付き → `restricted`
3. 判断に迷ったら `OK?`（疑問符付き）と書いて Claude Code に確認を頼んでください。

> ⚠️ ターミナル操作は **SOURCES の記入には不要**です。すべてブラウザで完結します。
> （集めたリストをサイトに反映する「アダプター作成」はエンジニア／Claude 側の仕事です。）

## 6. Telegram bot と VPS について（結論：今は使いません）

オーナーから質問のあった Telegram bot / VPS について、現状を正直に書きます。

### 6-1. このプロジェクトに Telegram bot は存在しません

- リポジトリ全体を調べましたが、**Telegram に関するコードは1つもありません**。
- このプロジェクトの「通知」の構想（`docs/HANDOFF.md` 第4章）は次の通りで、**Telegram は含まれていません**：
  - **メール通知**（チケット発売アラート、週刊ニュースレター）＝収益の柱（Pillar A/C）
  - **Web プッシュ通知**（PWA：スマホのホーム画面に追加できる仕組み）＝将来追加
- **したがって、Telegram bot を動かすためのターミナル操作も、現時点では必要ありません。**
- もし将来「Telegram でも通知したい」と思えば追加は可能ですが、それは**現構想の変更**になります。
  今回は構想を変えないご指示なので、**Telegram は使わない**という整理にします。

### 6-2. VPS（レンタルサーバー）も使いません

> 💡 用語：**VPS（ブイピーエス）**＝自分専用のレンタルサーバー。常時動かしたいプログラムを置く箱。

- このプロジェクトの自動データ取得は、**VPS を使わず**に動かす設計です。
- 定期実行（毎日〜2日に1回の巡回）は次のどちらかで行います（どちらも VPS 不要・追加費用ほぼ無し）：
  - **Vercel Cron**（本番サイトを動かしている Vercel の定期実行機能）
  - **GitHub Actions**（GitHub 上の自動実行。ここで第2章の GitHub Secrets を使う）
- **結論：VPS は使いません。** サーバーの契約・管理はオーナーには発生しません。
