0全体の地図（今日やること4つ）

オーナーの作業は大きく 4つ あります。順番が大事です。

① Supabase に SQL スキーマを流す
データの「箱」を作る（ブラウザのみ・コードなし）
↓
② 鍵（APIキー）を .env.local に書き込む
パソコンとデータベースをつなぐ
↓
③ ターミナルで seed → verify
箱の中にデータを入れて、正しいか確認
↓

💡 Supabase（スーパーベース）＝インターネット上にある「データの保管庫（データベース）」。公演データ（いつ・どこで・何の演目か）をここに貯めます。無料で始められます。

1Supabase に SQL スキーマを流す

1-1. そもそも「SQL スキーマ」とは？どこにある？

  • SQL（エスキューエル）＝データベースに指示を出すための言葉。
  • スキーマ＝「データをどんな形で保存するか」という設計図（テーブル＝表の定義）。
  • 探していたファイルはここにあります👇
supabase/migrations/
  ├── 001_initial_schema.sql   ← 古い版（使いません）
  ├── 002_rebuild_schema.sql   ← ★最初に流す（基本の箱）
  └── 003_entity_graph.sql    ← ★次に流す（横断検索の箱）
💡 migrations（マイグレーション）＝「データベースの設計変更ファイル」。番号順（001→002→003）に実行するのがルール。このプロジェクトでは 002 と 003 の2つを実行すれば OK。

1-2. 実行手順（ブラウザだけ・ターミナル不要）

  1. ブラウザで supabase.com にログインし、作成済みのプロジェクトを開く。
  2. 左のメニューから 「SQL Editor」（紙とペンのアイコン）をクリック。
  3. 右上の 「+ New query」 ボタンを押す。
  4. supabase/migrations/002_rebuild_schema.sql の中身を全部コピーして貼り付ける。
    💡 ターミナルで中身を表示したい場合：
    cat supabase/migrations/002_rebuild_schema.sql
  5. 緑色の 「Run」 ボタンを押す。
  6. 画面下に Success. No rows returned が出たら成功。
  7. 同じことを 003_entity_graph.sql でも繰り返す。
⚠️ 必ず 002 → 003 の順番で実行してください。逆にするとエラーになります。
赤いエラーが出たら、そのメッセージをそのまま Claude Code に貼って相談してください。

1-3. うまくいったかの確認

左メニューの 「Table Editor」 を開き、companies, performances, people, works, venues などのテーブルが並んでいれば成功です。

（この時点で中身が空っぽ＝0行でも正常です。データは第3章で入れます。）

2環境変数を .env.local に保存する

2-1. 「環境変数」「.env.local」とは？

  • 環境変数（かんきょうへんすう）＝プログラムに渡す「秘密の設定値」。今回は Supabase の鍵など。
  • .env.local＝その秘密を書いておくファイル。ファイル名がドット（.）で始まる。
  • local＝「自分のパソコンの中だけ」という意味。GitHub には絶対アップロードされません（安全）。
⚠️ 超重要：このファイルの中身（特に SERVICE_ROLE_KEY）は他人に絶対見せないこと。SNS・GitHub・スクショに写さない。漏れると DB を勝手に書き換えられます。

2-2. 何の鍵を入れるのか（3つ）

Supabase の 「Project Settings（歯車）→ API」 で取得できます。

環境変数の名前どこで取る役割
NEXT_PUBLIC_SUPABASE_URL「Project URL」DB の住所 公開OK
NEXT_PUBLIC_SUPABASE_ANON_KEY「anon public」読み取り用 公開OK
SUPABASE_SERVICE_ROLE_KEY「service_role」🔒書き込み用 極秘

2-3. .env.local を作る方法（コマンド付き）

💡 ターミナル＝パソコンに文字でコマンド（命令）を打ち込む画面。Mac は「ターミナル.app」、Windows は「PowerShell」または「コマンドプロンプト」。

方法A：テキストエディタ（かんたん・おすすめ）

VS Code やメモ帳で新規ファイルを作成し、ファイル名を .env.local にして、以下を貼り付けて保存：

NEXT_PUBLIC_SUPABASE_URL=https://あなたのプロジェクトID.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=ここにanonキーを貼る
SUPABASE_SERVICE_ROLE_KEY=ここにservice_roleキーを貼る
  • = の前後にスペースを入れない。
  • 値を引用符（" や '）で囲まない。
  • 1つのキーは必ず1行に収める（改行しない）。

方法B：ターミナルで作る

cat > .env.local << 'EOF'
NEXT_PUBLIC_SUPABASE_URL=https://あなたのプロジェクトID.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=ここにanonキーを貼る
SUPABASE_SERVICE_ROLE_KEY=ここにservice_roleキーを貼る
EOF

「ここに〜貼る」を本物のキーに書き換えてから実行してください。

確認コマンド：

cat .env.local

3行が表示されればOK。

2-4. GitHub Secrets の設定（将来必要になったとき）

💡 今すぐは不要です。将来 GitHub Actions（自動データ取得）を動かすときに使います。
  1. GitHubのリポジトリを開く（github.com/masatohagiwara484-ops/world-ballet-calendar）
  2. 上のタブの一番右 「Settings」（歯車） をタップ
  3. 左メニューを下にスクロールして 「Secrets and variables」 をタップ
  4. 「Actions」 を選ぶ
  5. 緑のボタン 「New repository secret」 を押す
  6. Name に変数名、Secret に値を貼る → 「Add secret」
  7. 3つ分（URL・ANON・SERVICE_ROLE）繰り返す

3seed ＋ verify（データを入れて確認する）

⚠️ ターミナル操作です。第1章（SQL）と第2章（.env.local）が先に終わっていることが条件です。

3-1. 用語の意味

  • seed（シード＝種まき）＝用意済みの公演データを Supabase の箱に流し込む作業。
  • verify / validate（検証）＝データに矛盾がないか自動チェックする作業。

3-2. 事前準備（初回だけ）

💡 Node.js（ノード）＝このプロジェクトを動かす土台のソフト。
npm（エヌピーエム）＝Node.js に付属する「部品の取り寄せ係」。

Node.js が入っているか確認：

node -v

v18 や v20 のような数字が出ればOK。
command not found と出たら nodejs.org から「LTS」版をインストール。

部品を取り寄せる（初回だけ）：

cd /home/user/world-ballet-calendar
npm install

数分かかります。完了したら次へ。

3-3. 実行コマンド（順番に）

① まずデータが正しいか検証（DBに触らない安全チェック）：

npm run validate:data

緑色で「問題なし」が出れば成功。赤いエラーが出たら内容を Claude Code に貼って相談。

② データを Supabase に投入（seed）：

npm run seed

「Seeded N companies / M performances」のようなメッセージが出れば完了。

③ サイトを開いて確認（任意）：

npm run dev

表示された http://localhost:3000 をブラウザで開く。終了は Ctrl + C。

3-4. よくあるエラーと対処

エラー内容原因対処
Missing Supabase env vars.env.local が無い第2章をやり直す
placeholders — refusing to seedキーが仮の値のまま本物のキーに差し替え
relation "companies" does not existSQL スキーマ未実行第1章の002→003を流す
command not found: npmNode.js 未インストール3-2 を実施
💡 次回からは①→②の2コマンドだけで更新できます。準備（3-2）は初回のみです。

4SOURCES リストの埋め方

docs/SOURCES.md の各団体に 4つの空欄 があります。コードは一切不要。調べて貼るだけです。

4-1. 各列に「何を貼るか」早見表

列の名前何を貼る？探し方
Listing URL
（公演一覧ページ）		 トップではなく「シーズン／演目一覧」の正確なURL 公式サイトで
Season・What's On・Calendar・公演情報 を探す
Feed
（機械が読める形式）		 自動取得できる URL。優先: API▸iCal▸RSS▸HTML 「カレンダー登録」「iCal」「RSS」ボタンを探す。無ければ HTML
Affiliate
（チケット販売）		 チケット購入URL または提携サービス名 「Book tickets」リンク。不明なら unknown
robots/ToS
（自動取得の可否）		 OK / restricted / forbidden 公式URL末尾に /robots.txt を付けてブラウザで開く

4-2. Feed（フィード）をやさしく説明

  • iCal（アイカル）＝カレンダーアプリに取り込める予定ファイル（.ics）。
  • RSS＝更新情報を配信する仕組み。
  • API＝データを直接くれる窓口。
  • Feedがあると、サイトの見た目が変わっても壊れません。最も重要な列です。

4-3. robots/ToS の確認方法（ブラウザのみ）

  1. 劇場の公式URLの末尾に /robots.txt を付けてブラウザで開く
    例：https://www.rbo.org.uk/robots.txt
  2. 表示されたテキストを見る：
    • 全部許可っぽい → OK
    • 公演ページが Disallow で禁止 → forbidden
    • 一部だけ条件付き → restricted
  3. 判断に迷ったら OK?（疑問符付き）と書いて Claude Code に確認してもらう。
⚠️ SOURCES 記入にターミナル操作は不要です。すべてブラウザで完結します。

5作業の渡し方

  1. docs/SOURCES_FILL_IN_JA.md（空欄記入シート）を開く。
  2. 各団体の ____ の部分に、第4章で調べたリンク・判定を書き込む。
    全部そろわなくてOK。分かったものから埋めて構いません。
  3. 埋めたシートの中身をコピーして Claude Code のチャットに貼り付け、こう伝える👇
「SOURCES を埋めました。アダプターを作って取り込み（ingestion）を進めてください」

あとは Claude（またはエンジニア）が各行を「アダプター」というプログラムに変換します。あなたは内容を承認するだけです。

💡 アダプター＝1つの劇場サイト専用の「読み取りプログラム」。
review queue（レビューキュー）＝自動取得したデータを公開前に人間が確認する一時置き場。日付ミスは信用を壊すので、公開前にあなたがチェックする設計です。

6Telegram bot / VPS について

📌 結論：今は使いません。現在の構想には含まれていません。

6-1. Telegram bot はプロジェクトに存在しません

リポジトリ全体を調査しましたが、Telegram に関するコードは1つもありません。

このプロジェクトの「通知」の構想は：

  • メール通知（チケット発売アラート、週刊ニュースレター）＝収益の柱
  • Web プッシュ通知（PWA：スマホのホーム画面に追加できる仕組み）＝将来追加予定

Telegram を動かすためのターミナル操作は、現時点では必要ありません。

6-2. VPS（レンタルサーバー）も使いません

💡 VPS（ブイピーエス）＝自分専用のレンタルサーバー。常時動かしたいプログラムを置く箱。

定期実行（毎日〜2日に1回の巡回）は VPS を使わず に動かす設計です。

  • Vercel Cron＝本番サイトを動かしている Vercel の定期実行機能（追加費用ほぼなし）
  • GitHub Actions＝GitHub 上の自動実行（ここで第2章の GitHub Secrets を使う）

結論：VPS は使いません。サーバーの契約・管理はオーナーには発生しません。

  • ③ npm install（初回だけ）（第3章）

  • ③ npm run seed でデータ投入 → Table Editor で確認（第3章）

  • ④ 埋めた内容を Claude Code に貼って「取り込み開始」と依頼（第5章）
🙋 困ったら、エラーメッセージや画面のスクショ（鍵が写らないように注意）をそのまま Claude Code に貼って質問してください。1つずつ一緒に解決します。
